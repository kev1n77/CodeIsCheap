09 · Release Operations

签名发布与回滚

发布只从已进入 main 的版本 tag 启动。Windows Authenticode、macOS Developer ID 与 notarization、Tauri updater 签名缺一不可。

失败关闭任何证书、私钥、平台签名、notarization、sidecar 签名、updater `.sig`、版本或哈希校验失败，都不得发布非草稿 Release。

发布链

main 中创建 v<semver> tag
→
签名 sidecar
→
签名并 notarize 桌面包
→
校验 updater 与哈希
→
草稿上传后发布

触发与证据

  1. 同步 tauri.conf.json、桌面 package.json 和 Tauri Cargo.toml 的版本。
  2. 确认 main CI 全绿后创建并推送 v<semver> tag，或手动选择已存在 tag 运行 Signed release。
  3. 审批 release environment；工作流会拒绝不在 main 中的 tag。
  4. 保存工作流链接、release-manifest.v0.1.json、latest.json、安装包和 `.sig` 作为发布证据。

回滚演练

  • 在干净 Windows 与 macOS 设备安装上一稳定版并写入已知测试捕获。
  • 从正式 Release 执行应用内更新，验证平台签名、updater 签名和版本切换。
  • 使用测试迁移故障构建触发主库打开失败，确认只读恢复、搜索与脱敏导出可用。
  • 确认主数据库未覆盖，系统代理已恢复，捕获、证书与二次更新控件均禁用。
  • 重新安装稳定版并恢复主库；记录版本、平台、步骤、日志代码、截图和结果。

真实回滚演练必须基于正式签名制品与隔离测试数据。仓库测试只能证明机制，不能替代发布负责人签署的设备证据。

发布附件

必须存在

  • Windows NSIS 安装包、updater archive 及 `.sig`
  • macOS DMG、app updater archive 及 `.sig`
  • latest.json
  • release-manifest.v0.1.json

必须验证

  • 版本 tag 与三处 manifest 一致
  • Windows Authenticode 为 Valid
  • macOS notarization ticket 已 stapled
  • 所有文件大小与 SHA-256 匹配
